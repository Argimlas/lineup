# Multi-festival tabs rework

Mode: Autonomous (commit after each task/cluster on a feature branch, no push).

Builds on top of existing uncommitted WIP (per-festival storage in `useLineup`/`useFestivals`,
presets manifest, example lineup) — only the switcher UI and Editor layout are being reworked.

## 1. Data layer
- [ ] `useFestivals.createFestival(name)`: cap at 4 festivals — return `null` and create nothing
      if `index.length >= 4`; otherwise prepend the new entry (`[{id,name}, ...prev]`) so it
      becomes the leftmost tab, and keep setting it active. Return type becomes `string | null`.
  - Success: creating a 5th festival returns `null` and leaves `festivals` unchanged; creating
    while under the cap adds the new entry at index 0.
- [ ] `useLineup`: add `setFestivalFor(id, festival)` that writes directly to the cache/storage
      for an arbitrary festival id (not just the currently-bound `festivalId`), so a newly
      created festival can be populated with preset data in the same handler without racing the
      `activeId` state update.
  - Success: calling `setFestivalFor('some-other-id', data)` while a different festival is
    active persists `lineup_some-other-id` correctly and doesn't touch the active festival's data.
- [ ] `useConsent`: remove the now-unused `reset` export (replaced by the full-wipe handler in
      task 5).
  - Success: `tsc`/build has no unused-export/reference errors.

## 2. Header: dropdown → tabs
- [ ] Delete `src/components/FestivalSwitcher.tsx`; add `src/components/FestivalTabs.tsx` — a
      pure tab bar (`festivals`, `activeId`, `onSelect` props only, no create/delete buttons),
      styled like the existing `.day-tab`/`.day-tab.active` pattern.
- [ ] Wire `FestivalTabs` into `App.tsx`'s `app-header-row`, to the right of the `<h1>Lineup</h1>`
      and before the Help button.
- [ ] Add `.festival-tab`/`.festival-tab.active`/`.festival-tabs-row` CSS (mirroring
      `.day-tab`), plus mobile wrapping consistent with the existing 640px breakpoint rules.
  - Success: with 1-4 festivals, header shows that many tabs and no dropdown; clicking a tab
    switches the active festival exactly as the old dropdown did; layout doesn't overlap the
    Help button at 375px width.

## 3. Editor: "Import festival" section
- [ ] Rename the `<summary>Lineup</summary>` to `<summary>Import/Edit</summary>`.
- [ ] Replace the old preset `<select>` + Load button with: an `<h3>Import festival</h3>`
      heading, then a tab/button row showing up to 4 presets from the manifest plus a
      "+ New festival" button.
- [ ] "+ New festival" button: prompts for a name (same as today), calls the new
      `onCreateFestival` prop; if it returns `null` (cap hit), `alert('You can only have up to
      4 festivals.')`.
- [ ] Clicking a preset tab: fetch + parse it immediately (no intermediate paste-box step).
      - If the active festival has no days yet, load straight into it via the existing
        `setFestival` prop.
      - If the active festival already has a lineup, open a 3-way confirm modal (styled like
        the existing help modal): **Override current** / **Load as new festival** / **Cancel**.
        - Override → `setFestival(parsed)` on the active festival.
        - New festival → `onCreateFestival(preset.name)`; if capped, alert and stop; otherwise
          `onSetFestivalFor(newId, parsed)`.
  - Success: preset tabs capped at 4 even if the manifest lists more; loading into an empty
    active festival needs no confirmation; loading over a populated one shows the 3-way modal
    and each option does what it says; hitting the 4-festival cap during "new festival" shows
    the alert and does not create a tab.

- [ ] Increase the spacing between the festival name row and this button row (currently

  - Success: only one destructive lineup-level button exists in the Editor; confirming it
